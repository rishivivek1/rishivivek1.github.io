    What GPT-5.4 Is Good At

    Think of GPT-5.4 as the most diligent student in class — but one who needs very clear instructions on the assignment sheet:

    • Stays in character — If you tell it to be formal, it won't randomly switch to casual halfway through
    • Doesn't give up easily — On multi-step tasks, it keeps going instead of stopping after step 2
    • Great at combining info — Give it 10 documents and it'll pull together a solid summary
    • Follows structured rules — The more organized your prompt, the better the output
    • Can multitask — It can look up multiple things at once without getting confused
    • Good with numbers — Spreadsheets, math, and data formatting are strong suits

    The 8 Prompt Tricks That Fix All of This

    1. Output Contract — "Here's Exactly What I Want Back"

    Imagine telling someone "write me an essay" vs "write me a 5-paragraph essay with an intro, 3 body paragraphs about climate change, and a conclusion." The second one always gets better results. That's what an output contract does.

    Prompt 
    <output_contract>
- Return exactly the sections requested, in the requested order.
- If the prompt defines a preamble, analysis block, or working section, do not treat it as extra output.
- Apply length limits only to the section they are intended for.
- If a format is required (JSON, Markdown, SQL, XML), output only that format.
</output_contract>

<verbosity_controls>
- Prefer concise, information-dense writing.
- Avoid repeating the user's request.
- Keep progress updates brief.
- Do not shorten the answer so aggressively that required evidence, reasoning, or completion checks are omitted.
</verbosity_controls>

    Why it works: Instead of hoping GPT gives you the right format, you're handing it a blueprint.

    Try this — Copy and paste into ChatGPT 
    <output_contract>
- Return exactly 3 sections: Summary, Key Findings, Action Items.
- Summary: max 3 sentences.
- Key Findings: bullet list, max 5 items.
- Action Items: numbered list with owner and deadline.
- Do not add any other sections.
</output_contract>

<verbosity_controls>
- No filler. No repeating my question back to me.
- Keep it dense and useful.
</verbosity_controls>

Here are my meeting notes from today's product review:
[paste your notes here]

    2. Follow-Through Policy — "Stop Asking, Just Do It"

    Ever had a friend who asks "Should I pass the salt?" when you're clearly reaching for it? That's GPT without this rule. It keeps asking for permission on obvious things. This tag tells it: if it's safe and reversible, just go ahead.

    Prompt 
    <default_follow_through_policy>
- If the user's intent is clear and the next step is reversible and low-risk, proceed without asking.
- Ask permission only if the next step is:
  (a) irreversible,
  (b) has external side effects (sending, purchasing, deleting, or writing to production), or
  (c) requires missing sensitive information or a choice that would materially change the outcome.
- If proceeding, briefly state what you did and what remains optional.
</default_follow_through_policy>

    Why it works: You get flow instead of a Q&A session.

    Try this — Copy and paste into ChatGPT 
    <default_follow_through_policy>
- If my intent is clear and the next step is reversible, just do it.
- Only ask me if something is irreversible, costs money, or you're missing critical info.
- After doing something, briefly tell me what you did.
</default_follow_through_policy>

I have a Python script that reads a CSV and outputs a chart.
The chart title is wrong, the axis labels are missing,
and the colors are ugly. Fix all of it.

    Without this tag, GPT would ask: "Should I fix the title first? What colors do you prefer? What labels do you want?" With it, GPT just fixes everything and tells you what it changed.

    3. Instruction Priority — "My Latest Instruction Wins"

    Mid-conversation, you might say "Actually, make it shorter" or "Switch to bullet points." Without this rule, GPT gets confused about which instruction to follow. This sorts it out.

    Prompt 
    <instruction_priority>
- User instructions override default style, tone, formatting, and initiative preferences.
- Safety, honesty, privacy, and permission constraints do not yield.
- If a newer user instruction conflicts with an earlier one, follow the newer instruction.
- Preserve earlier instructions that do not conflict.
</instruction_priority>

    If you need to change direction mid-chat, be specific about what's changing:

    Prompt 
    <task_update>
For the next response only:
- Do not complete the task.
- Only produce a plan.
- Keep it to 5 bullets.

All earlier instructions still apply unless they conflict with this update.
</task_update>

    Why it works: It's like telling a group project partner: "Ignore what I said in the morning, here's the new plan — but everything else stays the same."

    Try this — Copy and paste into ChatGPT

    Start a conversation with this system prompt:

    <instruction_priority>
- My latest instruction always wins over earlier ones.
- Safety and honesty rules never change.
- Keep everything else I said before unless I specifically override it.
</instruction_priority>

You are a marketing copywriter. Write in a casual, witty tone.
All outputs should be under 100 words.

    Then mid-conversation, say:

    <task_update>
For the next response only:
- Switch to formal tone.
- Ignore the 100-word limit — go up to 300 words.
Everything else still applies.
</task_update>

Write the product description for our enterprise security tool.

    GPT writes one formal 300-word response, then automatically goes back to casual and short for the next message.

    4. Tool Persistence — "Don't Skip Steps"

    This is GPT-5.4's biggest weakness. It sees the finish line and sprints past all the prep work. Imagine a student who writes the conclusion of a lab report without actually doing the experiment. This tag forces it to do every step.

    Prompt 
    <tool_persistence_rules>
- Use tools whenever they materially improve correctness, completeness, or grounding.
- Do not stop early when another tool call is likely to materially improve correctness or completeness.
- Keep calling tools until:
  (1) the task is complete, and
  (2) verification passes (see <verification_loop>).
- If a tool returns empty or partial results, retry with a different strategy.
</tool_persistence_rules>

<dependency_checks>
- Before taking an action, check whether prerequisite discovery, lookup, or memory retrieval steps are required.
- Do not skip prerequisite steps just because the intended final action seems obvious.
- If the task depends on the output of a prior step, resolve that dependency first.
</dependency_checks>

    When tasks don't depend on each other, let GPT do them at the same time:

    Prompt 
    <parallel_tool_calling>
- When multiple retrieval or lookup steps are independent, prefer parallel tool calls to reduce wall-clock time.
- Do not parallelize steps that have prerequisite dependencies or where one result determines the next action.
- After parallel retrieval, pause to synthesize the results before making more calls.
- Prefer selective parallelism: parallelize independent evidence gathering, not speculative or redundant tool use.
</parallel_tool_calling>

    Why it works: Forces GPT to "show its work" step by step instead of jumping to conclusions.

    Try this — Copy and paste into a coding agent (Cursor, Copilot, etc.) 
    <tool_persistence_rules>
- Use tools whenever they improve correctness.
- Don't stop until the task is complete AND verification passes.
- If a search returns nothing, try a different query before giving up.
</tool_persistence_rules>

<dependency_checks>
- Before making changes, check what files depend on the thing I'm changing.
- Don't skip the lookup just because the fix seems obvious.
</dependency_checks>

Rename the function `getUserData` to `fetchUserProfile` across
the entire codebase. Make sure every import, every test file,
and every reference is updated — not just the definition.

    Without these tags, GPT might rename the function definition and a few obvious references, then say "Done!" — missing 5 test files and 2 imports. With the tags, it searches everywhere first, builds the full list, then makes all the changes.

    5. Completeness Contract — "Don't Say Done Until It's Actually Done"

    Processing a list of 20 items? GPT might do 15 and call it a day. This tag is your insurance policy — it forces GPT to track progress like a checklist.

    Prompt 
    <completeness_contract>
- Treat the task as incomplete until all requested items are covered or explicitly marked [blocked].
- Keep an internal checklist of required deliverables.
- For lists, batches, or paginated results:
  - determine expected scope when possible,
  - track processed items or pages,
  - confirm coverage before finalizing.
- If any item is blocked by missing data, mark it [blocked] and state exactly what is missing.
</completeness_contract>

<empty_result_recovery>
If a lookup returns empty, partial, or suspiciously narrow results:
- do not immediately conclude that no results exist,
- try at least one or two fallback strategies,
  such as:
  - alternate query wording,
  - broader filters,
  - a prerequisite lookup,
  - or an alternate source or tool,
- Only then report that no results were found, along with what you tried.
</empty_result_recovery>

    Why it works: Like telling a student "Check off each question before submitting the test."

    Try this — Copy and paste into ChatGPT 
    <completeness_contract>
- Don't say you're done until every company on the list is covered.
- If you can't find data for one, mark it [blocked] and say what's missing.
- Track your progress as you go.
</completeness_contract>

<empty_result_recovery>
- If a search comes back empty, try different keywords before giving up.
- Try at least 2 alternate approaches.
</empty_result_recovery>

Research these 10 companies and give me each one's:
- Founded year
- CEO name
- Latest funding round
- Employee count (approximate)

Companies: Stripe, Notion, Linear, Vercel, Supabase,
Railway, Resend, Cal.com, Trigger.dev, Inngest

    Without the completeness contract, GPT might cover 7 out of 10 and stop. With it, GPT tracks each company like a checklist and flags any gaps.

    6. Verification Loop — "Double-Check Before Submitting"

    This is the most important pattern in the entire guide. Before GPT gives you the final answer or takes a real-world action (like sending an email), it runs through a quick self-check.

    Prompt 
    <verification_loop>
Before finalizing:
- Check correctness: does the output satisfy every requirement?
- Check grounding: are factual claims backed by the provided context or tool outputs?
- Check formatting: does the output match the requested schema or style?
- Check safety and irreversibility: if the next step has external side effects, ask permission first.
</verification_loop>

<missing_context_gating>
- If required context is missing, do NOT guess.
- Prefer the appropriate lookup tool when the missing context is retrievable; ask a minimal clarifying question only when it is not.
- If you must proceed, label assumptions explicitly and choose a reversible action.
</missing_context_gating>

    For actions that affect the real world (sending messages, updating databases, buying things):

    Prompt 
    <action_safety>
- Pre-flight: summarize the intended action and parameters in 1-2 lines.
- Execute via tool.
- Post-flight: confirm the outcome and any validation that was performed.
</action_safety>

    Why it works: It's the "re-read your answer before turning in the exam" rule — but for AI.

    Try this — Copy and paste into ChatGPT 
    <verification_loop>
Before giving me your final answer:
- Does it cover all 4 sections I asked for?
- Are all facts backed by sources you actually found (not made up)?
- Is the format exactly what I requested (markdown table)?
- If anything could have real-world consequences, flag it.
</verification_loop>

<missing_context_gating>
- If you're not sure about something, say so. Don't guess.
- Label any assumptions clearly.
</missing_context_gating>

Compare AWS Lambda vs Google Cloud Functions vs Azure Functions.
Give me a markdown table with: cold start time, max execution time,
free tier limits, supported languages, and pricing per million invocations.
Only use current 2026 pricing — don't guess old numbers.

    The verification loop forces GPT to self-audit before responding. If it can't find 2026 pricing for one service, it says "I couldn't verify this" instead of quietly making up a number.

    7. Research Mode — "Plan, Search, Then Write"

    When you need GPT to research a topic, don't just say "research X." Give it a 3-step process: first plan what to look for, then actually search, then combine everything into one answer.

    Prompt 
    <research_mode>
- Do research in 3 passes:
  1) Plan: list 3-6 sub-questions to answer.
  2) Retrieve: search each sub-question and follow 1-2 second-order leads.
  3) Synthesize: resolve contradictions and write the final answer with citations.
- Stop only when more searching is unlikely to change the conclusion.
</research_mode>

<citation_rules>
- Only cite sources retrieved in the current workflow.
- Never fabricate citations, URLs, IDs, or quote spans.
- Use exactly the citation format required by the host application.
- Attach citations to the specific claims they support, not only at the end.
</citation_rules>

<grounding_rules>
- Base claims only on provided context or tool outputs.
- If sources conflict, state the conflict explicitly and attribute each side.
- If the context is insufficient or irrelevant, narrow the answer or say you cannot support the claim.
- If a statement is an inference rather than a directly supported fact, label it as an inference.
</grounding_rules>

    Why it works: It's the difference between a student who Googles once and writes, vs one who makes an outline, reads 5 sources, and then writes. The second one always gets better grades.

    Try this — Copy and paste into ChatGPT (with web search enabled) 
    <research_mode>
Do this in 3 steps:
1) Plan: list 4-5 sub-questions you need to answer.
2) Search: look up each sub-question. Follow 1-2 promising leads.
3) Synthesize: combine everything into one answer with citations.
Stop only when more searching won't change your conclusion.
</research_mode>

<citation_rules>
- Only cite sources you actually found and read.
- Never make up a URL or reference.
- Put the citation next to the specific claim it supports.
</citation_rules>

<grounding_rules>
- If two sources disagree, tell me both sides.
- If you're not sure, say "this is my inference" — don't present it as fact.
</grounding_rules>

Research: What is the current state of humanoid robotics in manufacturing?
Who are the top 5 companies, what stage are they at,
and what's the realistic timeline for warehouse deployment?

    Instead of giving you one shallow summary, GPT plans its research, searches methodically, follows leads, and produces a cited, multi-source answer with clearly labeled inferences.

    8. Structured Output Lock — "Give Me Just the Data, Nothing Else"

    When you need GPT to output code, JSON, SQL, or any machine-readable format — it sometimes wraps it in explanations you didn't ask for. This locks it down.

    Prompt 
    <structured_output_contract>
- Output only the requested format.
- Do not add prose or markdown fences unless they were requested.
- Validate that parentheses and brackets are balanced.
- Do not invent tables or fields.
- If required schema information is missing, ask for it or return an explicit error object.
</structured_output_contract>

    Why it works: You asked for a spreadsheet, not a spreadsheet with a cover letter.

    Try this — Copy and paste into ChatGPT 
    <structured_output_contract>
- Output ONLY valid JSON. No explanations before or after.
- No markdown code fences.
- Don't invent fields I didn't ask for.
- If something is missing, return a JSON error object instead of guessing.
</structured_output_contract>

Convert this product list into JSON:

Name: Wireless Mouse, Price: $29.99, Stock: 145, Category: Electronics
Name: Standing Desk, Price: $449.00, Stock: 23, Category: Furniture
Name: USB-C Hub, Price: $39.99, Stock: 0, Category: Electronics
Name: Monitor Arm, Price: $89.99, Stock: 67, Category: Furniture

    Without this tag, GPT might say "Here's the JSON:" followed by an explanation, wrapped in markdown fences, and maybe add extra fields like id or createdAt that you never asked for. With the tag, you get clean, raw JSON — nothing else.

    The Reasoning Effort Dial — Less Is Often More

    Here's the thing nobody expects: making GPT "think harder" doesn't always make it smarter. OpenAI themselves say most people should stick to none through medium.

    Think of it like exam strategy. Spending 2 hours on a 5-mark question isn't smart — you'd be better off using that time across the whole paper.

    Thinking Level When to Use It Everyday Analogy
    none Quick tasks — sorting, labeling, simple formatting Answering "What's 2+2?"
    low Slightly complex tasks that need a small accuracy boost Solving a word problem in math
    medium Research, comparing sources, long summaries. The sweet spot. Writing an essay with multiple references
    high Tasks that genuinely need deep thinking and you're okay waiting longer Solving an Olympiad problem
    xhigh Super long, complex agent tasks. Don't use this by default. Writing a PhD thesis

    Before you crank up the thinking level, try these first: 1. Add a <completeness_contract> — forces it to finish everything 2. Add a <verification_loop> — forces it to double-check 3. Add <tool_persistence_rules> — forces it to do all the steps

    Still getting shallow answers? Add this nudge:

    Prompt 
    <dig_deeper_nudge>
- Don't stop at the first plausible answer.
- Look for second-order issues, edge cases, and missing constraints.
- If the task is safety or accuracy critical, perform at least one verification step.
</dig_deeper_nudge>

    Bonus: Coding-Specific Prompts

    If you're using GPT-5.4 to write or fix code, these two blocks make a big difference.

    "Don't just analyze — actually fix it":

    Prompt 
    <autonomy_and_persistence>
Persist until the task is fully handled end-to-end within the current turn whenever feasible: do not stop at analysis or partial fixes; carry changes through implementation, verification, and a clear explanation of outcomes unless the user explicitly pauses or redirects you.

Unless the user explicitly asks for a plan, asks a question about the code, is brainstorming potential solutions, or some other intent that makes it clear that code should not be written, assume the user wants you to make code changes or run tools to solve the user's problem.
</autonomy_and_persistence>

    "Use the right tool, not the terminal, for everything":

    Prompt 
    <terminal_tool_hygiene>
- Only run shell commands via the terminal tool.
- Never "run" tool names as shell commands.
- If a patch or edit tool exists, use it directly; do not attempt it in bash.
- After changes, run a lightweight verification step such as ls, tests, or a build before declaring the task done.
</terminal_tool_hygiene>

    The One Line That Sums It All Up

    GPT-5.4 is the smartest model OpenAI has released — but the biggest improvement doesn't come from making it think harder. It comes from telling it what "done" looks like.

    Define the output. Add a self-check. List what "complete" means. Then worry about reasoning effort.

    That's the entire playbook.

    Tag Cheat Sheet — Quick Reference

    GPT-5.4 works best when you wrap your instructions inside XML tags — think of them like labeled boxes. You put rules inside a box, slap a name on it, and GPT-5.4 knows exactly what those rules are for.

    Copy any tag block from this guide, paste it into your prompt (at the start works best), and GPT-5.4 will follow those rules for the rest of the conversation.

    Tag What It Tells GPT-5.4
    <output_contract> "Here's the exact format and sections I want back."
    <verbosity_controls> "Keep it short and dense — no fluff."
    <default_follow_through_policy> "Just do it if it's safe. Only ask me if it's risky."
    <instruction_priority> "If my new instructions clash with old ones, follow the new ones."
    <task_update> "I'm changing direction — here's what's different now."
    <tool_persistence_rules> "Don't stop using tools until the job is truly done."
    <dependency_checks> "Check what you need before you act — don't skip steps."
    <parallel_tool_calling> "Run independent lookups at the same time to save time."
    <completeness_contract> "Don't say you're done until everything on the list is covered."
    <empty_result_recovery> "If a search comes back empty, try a different approach before giving up."
    <verification_loop> "Double-check your work before handing it to me."
    <missing_context_gating> "If you're missing info, look it up — don't guess."
    <action_safety> "Before doing anything in the real world, tell me what you're about to do."
    <research_mode> "Do research in 3 rounds: plan, search, then combine findings."
    <citation_rules> "Only cite real sources. Never make up links or references."
    <grounding_rules> "Only say things you can back up with actual evidence."
    <structured_output_contract> "Give me raw code/data only — no explanations around it."
    <dig_deeper_nudge> "Don't stop at the first answer — look for edge cases."
    <autonomy_and_persistence> "Keep working until the task is fully done, don't stop halfway."
    <terminal_tool_hygiene> "Use the right tool for the right job — don't mix them up."

    Where It Still Messes Up

    Even the smartest student makes mistakes without clear instructions:

    • Picks the wrong tool early on — Like grabbing a calculator when you need a dictionary
    • Skips steps it thinks are "obvious" — Jumps to the answer without doing the work first
    • More thinking ≠ better answers — Sometimes overthinking makes it worse (seriously)
    • Makes up sources — If you don't tell it to only use real references, it might invent them
    • Acts before asking — It'll delete something without checking with you first
    • Gets confused in code editors — Needs clear rules about which coding tool to use
